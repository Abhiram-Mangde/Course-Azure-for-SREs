# Contributing to Azure for SRE Course

Thank you for your interest in contributing to the Azure for SRE course! Your contributions will help improve this resource and make it more helpful for everyone learning Site Reliability Engineering (SRE) with Azure.

*Please take a moment to review this guide to ensure a smooth contribution process.*

## How to Contribute
We welcome contributions in the form of:
- Bug Reports
- Enhancements
- Tutorials
- Examples
- Fixing typos or improving clarity
- General feedback

### 1. Fork the Repository
To get started, fork the repository to your own GitHub account:
1. Navigate to the Azure for SRE course repository on GitHub.
2. Click the "Fork" button at the top-right corner of the page.
3. This creates a copy of the repository in your GitHub account.

### 2. Clone Your Fork
Clone the repository to your local machine to start working:
```bash
git clone https://github.com/YOUR-USERNAME/azure-for-sre.git
```

### 3. Create a New Branch
Create a new branch for your contribution:
```bash
git checkout -b your-branch-name
```
Make sure your branch name is descriptive, and try to include the purpose of the change.

### 4. Make Changes
Make your changes! Whether you're:
- Updating documentation
- Adding new content
- Correcting existing content
- Enhancing examples
- Be sure to test your changes if applicable.

### 5. Commit Your Changes
Commit your changes with a clear, concise message:
```bash
git add .
git commit -m "Descriptive commit message"
```
Ensure your commit message follows this format:
- **Bug fix:** "Fix issue with X in Y section"
- **Enhancement:** "Add new tutorial on Z"
- **Docs:** "Improve explanation for X in the VM section"

### 6. Push Changes to Your Fork
Push your changes to your forked repository on GitHub:
```bash
git push origin your-branch-name
```

### 7. Create a Pull Request (PR)
Go to the Pull Requests tab in the main repository and click "**New Pull Request**"
- Select your branch from your fork and compare it with the main branch of the original repository.
- Provide a clear description of what changes you've made and why they improve the course content.
- Click "Create Pull Request" to submit your changes for review.

## Pull Request Guidelines
To ensure that your pull request is accepted and reviewed efficiently, please follow these guidelines:
- **Be clear:** Provide a detailed description of what your PR does. If it addresses an issue, reference the issue number (e.g., Closes #5).
- **Keep it concise:** Focus on one topic per pull request. If you're making several changes, consider splitting them into separate PRs.
- **Code of Conduct:** Please be respectful and kind in all your interactions. We strive to maintain a positive, inclusive, and friendly community.
- **Follow the style:** Make sure your changes align with the style and format used in the repository.
  - **For documentation:** Use simple, clear language. Avoid jargon where possible.
  - **For examples:** Ensure examples are easy to follow and work correctly.

## Content Guidelines
Here are a few guidelines to follow when adding new content to the course:

1. Clarity and Simplicity: Our goal is to make the course accessible and easy to understand. Please write in a way that someone new to Azure and SRE can follow.
2. Accuracy: Ensure your content is accurate, up-to-date, and based on the latest Azure features and practices.
3. Practicality: We aim to provide hands-on learning experiences. Where possible, include practical examples, real-world use cases, and clear, executable steps.
4. Avoid Overly Complex Explanations: While advanced topics are welcome, try to break down concepts and avoid overwhelming readers with too much technical jargon or overly complex explanations.
5. Link to External Resources: If relevant, link to official documentation, articles, or tutorials that can offer deeper explanations.

## Issues and Discussion
If you're not sure how to contribute, or you have ideas for improvements, feel free to open an issue:
1. Go to the Issues tab in the repository.
2. Click "New Issue" to describe any bugs, problems, or ideas you have for enhancing the course content.

We encourage discussions, ideas, and feedback. If you spot something that could be improved, don't hesitate to start a discussion or raise an issue.
